# 运行时验证 (R1-R4)

> 静态分析只证明"代码看起来对"，运行时验证证明"代码实际能跑"。
> 在静态分析（F*/U*）之后执行。

---

## R1 — 构建验证

执行目标项目的构建命令：
- LLM 自行检测构建方式（读 package.json/go.mod/pubspec.yaml/Makefile 等）
- 构建失败 → 直接标记为 gap，附编译错误信息
- 评分：通过=100，失败=0

## R2 — 冒烟启动

尝试启动目标应用（如果项目可独立启动）。LLM 根据项目类型选择验证方式：
- **Web 后端**：启动 server → 检查 health endpoint 返回 200
- **Web 前端**：启动 dev server → 检查首页可访问
- **移动端**：在模拟器/设备上启动 → 检查进程存活 >5 秒
- **桌面端（WPF/MAUI/Electron）**：启动 exe → 检查进程存活 >5 秒 + 不输出 fatal 错误日志
- 启动失败（进程立即退出或报错）→ 标记 gap，附错误日志
- 无法独立启动（需要外部服务）→ 标记 `SMOKE_SKIPPED`，不扣分
- 评分：通过=100，失败=0，跳过=N/A

## R3 — 测试向量验证

**适用条件**：test-vectors.json 存在

跑 test-vectors.json 中的输入输出对：
- 对每个 vector：调用目标代码的对应函数 → 比对输出
- LLM 生成一个临时测试脚本执行 vectors
- 通过 → 该组件行为与源码一致
- 失败 → 标记精确的不一致点
- 评分 = 通过的 vector 数 / 总 vector 数 × 100

## R4 — 协议兼容性测试

**适用条件**：stack-mapping 中有 `compatibility: exact` 的通信组件且服务端可达

对 exact 通信组件：
- 生成一个最小客户端测试脚本
- 向源码配置的服务端发送真实请求
- 验证：连接成功、认证通过、数据格式正确
- 服务端不可达 → 标记 `PROTOCOL_SKIPPED`，不扣分
- 评分 = 通过的协议数 / 可测协议数 × 100

---

## 综合评分公式

```
静态分 = (有效 F* 维度 + 有效 U* 维度之和) / 有效维度数
运行时分 = (有效 R* 维度之和) / 有效运行时维度数
综合分 = 静态分 × 0.5 + 运行时分 × 0.5
```

- 有效 = 产物存在且总数 > 0
- N/A 维度不计入分母
- **综合分 < threshold 时，优先修运行时问题**（R1 失败 = 代码不能编译 = 一切白费）
- 视觉还原度由独立命令 `/cr-visual` 处理，不在 cr-fidelity 中
