# Feature Prune — 执行流程与规范总览

> 本文档是 feature-prune 技能的中枢文档，定义执行流程、模式、交互规范和词汇纪律。

---

## 1. 执行流程总览

```
Step 0: 项目画像 + 功能收集
  │  检测技术栈、收集计划功能 + 已实现功能、合并为全量清单
  │  输出: feature-list.json
  │  ▸ 用户确认功能清单完整
  ▼
Step 1: 核心场景识别
  │  从文档/代码识别用户场景、建立功能-场景映射、识别孤立功能
  │  输出: scenarios.json
  │  ▸ 用户确认场景划分
  ▼
Step 2: 必要性评估
  │  三维评估（场景关联、依赖程度、复杂度比）→ CORE / DEFER / CUT
  │  输出: assessment.json
  │  ▸ 用户逐个确认分类
  ▼
Step 3: 竞品参考（可选）★ quick 模式跳过
  │  WebSearch 3-5个同类竞品，提取功能范围，交叉对照
  │  输出: benchmark.json
  │  ▸ 用户确认是否采纳
  ▼
Step 4: 剪枝报告
  │  汇总所有结果，生成报告 + 行动清单
  │  输出: prune-report.md / prune-actions.json / prune-decisions.json
  ▼
  Done
```

---

## 2. 执行模式

| 模式 | 参数 | 执行步骤 | 适用场景 |
|------|------|----------|----------|
| **full** | 无参数 或 `full` | Step 0 → 1 → 2 → 3 → 4 | 首次剪枝、完整评估 |
| **quick** | `quick` | Step 0 → 1 → 2 → 4（跳过 Step 3） | 快速剪枝，不需要竞品参考 |
| **scope** | `scope <场景名>` | 加载已有 scenarios.json → 筛选指定场景 → Step 2 → 3 → 4 | 只审指定场景关联的功能 |

### 模式前置条件

```
full / quick     → 无前置条件，从零开始
                   若已有 prune-decisions.json → 自动复用用户历史决策

scope            → 必须已有 .allforai/feature-prune/scenarios.json
                   否则提示: "请先运行 /feature-prune full 生成场景数据"
```

## 3. 交互模式

### 核心原则：每个 Step 必须有用户确认环节

Claude 不得在用户确认前自动进入下一个 Step。每个 Step 的确认环节是**强制性的**，不可跳过。

### 确认检查点

| Step | 确认内容 | 询问方式 |
|------|----------|----------|
| Step 0 | 功能清单是否完整 | "我收集到以下 {N} 个功能（计划 {X} 个 + 已实现 {Y} 个 + 两者都有 {Z} 个）。是否有遗漏？是否需要排除某些功能？" |
| Step 1 | 场景划分是否合理 | "我识别出以下 {N} 个核心场景，另有 {M} 个功能无场景归属。场景划分合理吗？有遗漏的场景吗？" |
| Step 2 | 必要性分类是否正确 | "以下功能初步分类为：{CORE: X, DEFER: Y, CUT: Z}。请逐条确认或修改分类。" |
| Step 3 | 竞品参考是否采纳 | "竞品对照结果如下：{矩阵}。是否采纳这些参考信号？" |
| Step 4 | 最终报告是否准确 | 在对话中直接输出完整摘要，包含具体功能列表和分类详情 |

### 交互流程示意

```
Claude: [执行 Step N]
Claude: "Step N 完成。结果如下：..."
Claude: "请确认以上结果，或指出需要调整的地方。"
        ← 等待用户回复 →
用户:   "F-015 分类不对，应该是 DEFER 不是 CUT"
Claude: "已更新 F-015 为 DEFER，并记录理由。确认继续下一步？"
用户:   "继续"
Claude: [执行 Step N+1]
```

**禁止行为**：
- 不得将多个 Step 合并执行后一次性展示
- 不得在用户未回复时自动跳到下一个 Step
- 不得用 "如果没问题我继续了" 然后不等回复就继续

## 4. 词汇规范

### 允许使用的措辞

| 措辞 | 适用场景 |
|------|----------|
| "此功能无明确场景归属" | 功能未映射到任何核心场景（孤立功能） |
| "竞品在同阶段普遍不包含此功能" | 竞品对照显示覆盖率低（Step 3） |
| "实现复杂度较高，涉及 X 个文件" | 复杂度比评估中，量化描述复杂度 |
| "建议分类为 DEFER" | 给出初步分类建议，等待用户确认 |
| "场景关联度为弱" | 功能与核心场景的关联分析结果 |
| "无其他功能依赖此功能" | 依赖程度为 low 的事实陈述 |

### 绝对禁止的措辞

| 禁止措辞 | 违规原因 |
|----------|----------|
| "此功能不应该存在" | 做了价值判断 — 剪枝只分析必要性 |
| "你不需要这个功能" | 替用户做了决定 — 最终决定权归用户 |
| "这个功能过度设计了" | 主观评价 — 应该陈述客观维度评估 |
| "必须延后" | 强制口吻 — 应该用"建议分类为 DEFER" |
| "建议添加XX功能" | 违反"只剪不加"铁律 — 竞品有本项目没有也只标注不建议 |
| "需要补充XX" | 暗示当前不完整 — 剪枝不做添加建议 |

### 原则

剪枝只分析 **"必要性"**，不做 **"价值判断"**。

```
正确: "F-015(数据可视化大屏) 无明确场景归属，实现涉及 echarts + WebSocket 共 12 个文件，建议分类为 CUT。"
错误: "数据可视化大屏过度设计了，不应该存在。"

正确: "竞品在同阶段普遍不包含'数据可视化大屏'功能（0/4 覆盖）。"
错误: "你不需要数据可视化大屏，竞品都没做。"

正确: "竞品 3/4 包含'甘特图'功能，本项目暂未规划。仅标注，不纳入建议范围。"
错误: "竞品都有甘特图，建议添加。"
```

## 5. 输出文件

所有输出文件位于项目根目录下的 `.allforai/feature-prune/` 目录：

| 文件 | 产出 Step | 内容 | 格式 |
|------|-----------|------|------|
| `feature-list.json` | Step 0 | 全量功能清单（计划 + 已实现 + 合并标注来源） | JSON |
| `scenarios.json` | Step 1 | 核心场景列表 + 功能-场景映射 + 孤立功能标识 | JSON |
| `assessment.json` | Step 2 | 每个功能的三维评估 + CORE/DEFER/CUT 分类 + 证据 | JSON |
| `benchmark.json` | Step 3 | 竞品功能范围 + 覆盖率矩阵 + 对照信号 | JSON |
| `prune-report.md` | Step 4 | 人类可读的剪枝报告 | Markdown |
| `prune-actions.json` | Step 4 | 行动清单（CUT 如何处理、DEFER 推荐时机） | JSON |
| `prune-decisions.json` | Step 4 | 用户决策日志（scope 运行复用） | JSON |

### 目录结构

```
your-project/
└── .allforai/feature-prune/
    ├── feature-list.json       # Step 0 产出：全量功能清单
    ├── scenarios.json           # Step 1 产出：核心场景映射
    ├── assessment.json          # Step 2 产出：必要性评估
    ├── benchmark.json           # Step 3 产出：竞品参考（quick 模式无此文件）
    ├── prune-report.md          # Step 4 产出：剪枝报告
    ├── prune-actions.json       # Step 4 产出：行动清单
    └── prune-decisions.json     # Step 4 产出：用户决策日志（跨运行持久化）
```

## 6. 场景分析是核心

> Step 1（核心场景识别）是 feature-prune 技能的**核心价值所在**。没有场景锚点，剪枝就是主观意见；有了场景锚点，剪枝就是有据可依的决策。

### 为什么场景分析是核心

功能清单本身是扁平的 — 一串名字排在那里，看不出谁重要谁多余。场景分析把扁平清单变成**结构化的必要性地图**：每个功能都被拉到一个具体的用户场景下审视，回答"这个功能在哪个场景里被需要？"这个问题。

```
没有场景分析:
  20 个功能 → 凭感觉判断 → "我觉得这个不需要" → 主观意见

有了场景分析:
  20 个功能 → 映射到 5 个核心场景 → 3 个功能无场景归属 → 有据可依的 CUT 候选
                                   → 2 个功能仅弱关联 → 有据可依的 DEFER 候选
```

### 具体示例：订单管理场景

假设一个电商后台系统，识别出核心场景"商家处理客户订单"：

```
场景: 商家处理客户订单 (S-003)
  描述: 商家查看、处理、跟踪客户订单的完整流程
  角色: 商家管理员

功能清单中与"订单"相关的功能:
  ┌─────────────────────┬──────────┬──────────┬─────────────────────────────────┐
  │ 功能                │ 场景关联 │ 初步分类 │ 理由                            │
  ├─────────────────────┼──────────┼──────────┼─────────────────────────────────┤
  │ F-010 订单列表      │ core     │ CORE     │ 场景入口，必需                  │
  │ F-011 订单详情      │ core     │ CORE     │ 处理订单的核心操作页            │
  │ F-012 订单导出      │ core     │ CORE     │ 商家每月需要导出对账            │
  │ F-013 订单归档      │ related  │ DEFER    │ 关联但非当前阶段必需            │
  │ F-014 订单数据大屏  │ orphan   │ CUT      │ 无明确场景归属，复杂度高        │
  └─────────────────────┴──────────┴──────────┴─────────────────────────────────┘
```

分析过程：

```
F-012 订单导出:
  场景关联: strong — 商家处理订单场景的核心环节（对账需要导出）
  依赖程度: low — 无其他功能依赖
  复杂度比: low — 1 个 API 端点 + 1 个按钮
  → 虽然依赖低、复杂度低，但场景关联强 → CORE

F-013 订单归档:
  场景关联: medium — 与订单管理相关，但非日常操作流程
  依赖程度: low — 无其他功能依赖
  复杂度比: medium — 涉及归档策略、定时任务、存储迁移
  → 场景关联中等 + 复杂度中等 → DEFER（核心场景稳定后再实现）

F-014 订单数据大屏:
  场景关联: weak — 与"商家处理客户订单"场景无直接关系
  依赖程度: low — 无其他功能依赖
  复杂度比: high — 涉及 echarts、WebSocket 实时推送、数据聚合管道
  → 无明确场景归属 + 复杂度高 → CUT 候选
```

**没有场景分析，"订单导出"和"订单数据大屏"看起来都是"订单相关功能"，难以区分必要性；有了场景分析，前者是核心场景的一环，后者是无归属的孤立功能。**

## 7. 5 条铁律速查

| # | 铁律 | 一句话 | 违规示例 |
|---|------|--------|----------|
| 1 | **用户定性** | CORE/DEFER/CUT 最终分类由用户决定 | ~~自动将功能标为 CUT 不询问~~ |
| 2 | **只剪不加** | 只讨论"该不该做"，不建议"还应加什么" | ~~"竞品有甘特图，建议添加"~~ |
| 3 | **源码绑定** | 每条评估必须引用 `file:line` | ~~"此功能复杂度高"（没有文件引用）~~ |
| 4 | **场景为锚** | 必要性以用户场景为锚，不以技术优雅性为标准 | ~~"这个功能架构上不优雅，建议移除"~~ |
| 5 | **可逆标记** | CUT 是标记建议非删除代码，DEFER 标注推荐时机 | ~~直接建议删除代码文件~~ |

### 速记口诀

```
用户定性 — 分类用户说了算
只剪不加 — 只做减法不做加法
源码绑定 — 无证据不发言
场景为锚 — 场景定必要性
可逆标记 — 标记不是删除
```

## 8. 决策持久化机制

### prune-decisions.json 的作用

`prune-decisions.json` 保存用户在剪枝过程中做出的所有决策，使 `scope` 模式和后续运行不需要重复询问。

### 持久化的决策类型

```json
{
  "_meta": {
    "version": "1.0",
    "created_at": "2026-02-24T10:00:00Z",
    "updated_at": "2026-02-24T14:30:00Z",
    "project_hash": "abc123"
  },
  "decisions": [
    {
      "feature_id": "F-015",
      "feature_name": "数据可视化大屏",
      "suggested": "CUT",
      "final": "DEFER",
      "reason": "客户明确要求过，先延后观察",
      "decided_at": "2026-02-24T10:15:00Z"
    },
    {
      "feature_id": "F-013",
      "feature_name": "订单归档",
      "suggested": "DEFER",
      "final": "DEFER",
      "reason": "确认延后，核心场景稳定后实现",
      "decided_at": "2026-02-24T10:20:00Z"
    }
  ],
  "scenario_overrides": [
    {
      "feature_id": "F-018",
      "feature_name": "系统日志查看",
      "original_scenario": "orphan",
      "override_scenario": "S-002",
      "reason": "用户确认：属于系统管理场景"
    }
  ],
  "benchmark_overrides": [
    {
      "feature_id": "F-015",
      "signal": "strengthen_cut",
      "action": "ignore",
      "reason": "用户确认：不采纳竞品参考，客户已明确要求此功能"
    }
  ]
}
```

### scope 模式复用流程

```
/feature-prune scope 订单管理
        │
        ▼
  加载 .allforai/feature-prune/scenarios.json
  筛选"订单管理"场景关联的功能
        │
        ▼
  加载 prune-decisions.json
  （已有决策的功能 → 展示历史决策，询问是否沿用）
  （无决策的功能 → 正常执行 Step 2 评估）
        │
        ▼
  执行 Step 2 → Step 3 → Step 4（仅限筛选出的功能）
        │
        ▼
  更新 prune-decisions.json（合并新决策）
```

### 决策复用规则

| 决策类型 | 复用条件 | 失效条件 |
|----------|----------|----------|
| 分类决策（CORE/DEFER/CUT） | 无条件复用 | 用户手动修改 |
| 场景归属覆写 | 无条件复用 | 用户手动取消 |
| 竞品信号覆写 | 无条件复用 | 用户手动取消 |
| 评估维度确认 | 对应代码未变动时复用 | 对应代码有 git diff 变动 |

### 首次运行 vs scope 运行

```
首次运行 (full):
  ✅ 全量收集功能、识别场景、评估所有功能
  ✅ 每条分类都需要用户确认
  ✅ 生成 prune-decisions.json 作为基线

scope 运行 (scope <场景名>):
  ✅ 只处理指定场景关联的功能
  ✅ 自动复用已确认的决策（不重复询问）
  ✅ 仅对新发现或未决策的功能询问用户
  ✅ 合并更新 prune-decisions.json
```
