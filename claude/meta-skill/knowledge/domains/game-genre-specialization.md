# Game Genre Specialization Methodology

## Principle

Global bundled game sub-skills are for reusable production capabilities:
art direction, asset registry, UI layout, level difficulty budgets, runtime
import checks, audio contracts, VFX contracts, and other methods that apply
across many game types.

Do not create a global child skill for every genre, subgenre, or project. When
the important method depends tightly on one game's genre rules, generate a
project-local specialized skill during bootstrap.

## When To Specialize

Generate project-local specialized skills when at least one condition is true:

- The genre has an executable rule system that generic QA cannot validate.
- The level/content generator needs a genre-specific solver.
- Visual readability depends on a genre-specific play surface.
- Asset generation depends on a project-specific visual family, play surface,
  or screenshot context that generic art prompts cannot validate.
- Balance depends on genre-specific probability, timing, or combinatorics.
- Runtime code already contains a domain algorithm that must match design docs.
- A project-specific content taxonomy is too detailed for generic skills.

Do not specialize when a generic bundled skill can express the requirement with
parameters. Prefer parameters for broad concerns such as palette, platform,
engine, asset source, UI density, performance budget, or monetization style.

## Output Location

Write generated project-local specialized skills under:

```text
.allforai/bootstrap/specialized-skills/<specialization_id>/SKILL.md
```

Node-specs may read these files directly. They are project artifacts, not
installed global plugin skills, and Sub-Skill Mapping must not reference them.

## Required Structure

Every generated specialized skill must include:

- `Input Contract`
- `Output Contract`
- `Invocation Contract`
- `Automatic Validation`
- `Repair Routing`
- `Completion Conditions`

It must also include:

- specialization trigger evidence;
- project artifact paths it consumes;
- runtime code paths it must validate against, when applicable;
- generated artifact paths under `.allforai/`;
- blocked/failed states for missing executable validation;
- repair routing back to design, art, level, frontend, or tooling owners.

## Art Generation Specialization

Generate a project-local art generation specialization when the project's art
quality depends on a concrete play surface or asset family that cannot be
validated one asset at a time.

Typical triggers:
- dense board tiles where identity depends on the whole tile family;
- cards where frame, rarity, illustration, text zones, and hand readability
  must be validated together;
- rhythm notes/lanes where timing readability depends on scrolling layout;
- tower-defense units/projectiles where battlefield readability depends on
  team, lane, range, and VFX overlap;
- merge objects where chain progression must remain visually ordered;
- store screenshots or ad creatives where the first-viewport visual promise is
  more important than isolated asset quality.

The specialized art skill should define:
- asset family generation plan;
- project-specific prompt templates and negative prompts;
- model profile preferences from the global image model registry;
- local source search/adaptation rules when project assets can be reused;
- screen or board preview compositions for validation;
- visual-model/LLM review criteria for family distinctiveness and context fit;
- repair loop back to image generation, source search, adaptation, or art
  direction.

Suggested output path:

```text
.allforai/bootstrap/specialized-skills/<specialization_id>-art-generation/SKILL.md
```

Suggested project outputs:

```text
.allforai/game-design/art/specialized/<specialization_id>/art-generation-plan.json
.allforai/game-design/art/specialized/<specialization_id>/prompt-templates.json
.allforai/game-design/art/specialized/<specialization_id>/preview-validation-report.json
```

The specialized art skill must still use the global art contracts for source
strategy, image model routing, image generation, accepted-image manifests,
style QA, atlas/import QA, and runtime handoff. It only adds project-specific
planning, prompting, preview validation, and repair criteria.

## Generation Template

```markdown
---
name: project-specialized-<genre-or-method>
description: Project-local specialized skill generated by bootstrap for <project>.
---

# <Specialized Skill Name>

## Specialization Trigger

List the exact fields, documents, or runtime files that prove this project
needs the specialization.

## Input Contract

List required project artifacts and runtime files. Missing required inputs must
return `UPSTREAM_DEFECT` or `blocked_by_missing_input`.

## Output Contract

Write project-local `.allforai/` artifacts. Include reports, findings, metrics,
and repaired specs where relevant.

## Invocation Contract

Declare modes, input paths, output root, and runtime/tool requirements.

## Automatic Validation

Define deterministic checks first, then executable/runtime checks. Do not use a
weaker fallback when the specialization requires executable evidence.

## Repair Routing

Route failures back to the owning project node or artifact.

## Completion Conditions

State exact `COMPLETED`, `FAILED_VALIDATION`, and blocked conditions.
```

## Genre Examples

### Path-Connection Pair Matching

Applies to 连连看, Onet, Mahjong Connect, and similar games where the player
connects matching tiles through empty-space paths.

Typical generated specialized skills:
- tile-family readability against active board pools;
- board path-feedback readability for selected, hint, invalid, combo, and
  obstacle states;
- solver QA for max-turn path rules, edge traversal, portal rules, deadlocks,
  shuffle pressure, hint pressure, and difficulty bands.

Required project inputs usually include puzzle spec, tile manifest, board UI
layout, runtime matcher code, level data, and difficulty budget.

### Rhythm Timing

Applies when success depends on beat windows, latency, hit grading, or chart
readability.

Typical generated specialized skills:
- chart timing-window validation;
- input latency calibration QA;
- note density and readability QA;
- audio/chart alignment validation.

### Deck-Builder/Card Pool

Applies when success depends on draw probabilities, card synergies, rarity
curves, deck evolution, or combo loops.

Typical generated specialized skills:
- card-pool closure and legality checks;
- draw-probability and opening-hand validation;
- combo-loop exploit QA;
- deck archetype coverage QA.

### Roguelike/Procedural Map

Applies when generated levels must be reproducible, connected, and progression
safe.

Typical generated specialized skills:
- seed reproducibility validation;
- map connectivity and critical-path validation;
- reward/risk distribution QA;
- run progression softlock QA.

### Match-3 / Swap Puzzle

Applies when board solvability, cascades, special-piece interactions, gravity,
or shuffle pressure determine level quality.

Typical generated specialized skills:
- board-generator solvability validation;
- cascade and special-piece interaction simulation;
- move-count and target difficulty QA;
- dead-board and forced-shuffle pressure validation;
- visual board readability under cascade/VFX states.

Required project inputs usually include board rules, piece taxonomy, special
piece specs, level data, gravity/refill rules, target difficulty bands, and
runtime matcher/simulator code when available.

### Tower Defense / Path Defense

Applies when enemy pathing, tower coverage, wave pressure, resource pacing, and
map topology determine level quality.

Typical generated specialized skills:
- path connectivity and leak-route validation;
- tower coverage and blind-spot QA;
- wave DPS/EHP pressure simulation;
- economy pacing and build-order validation;
- enemy archetype counterplay coverage QA.

Required project inputs usually include path graph, enemy waves, tower stats,
placement rules, economy income, target fail/pass thresholds, and runtime
simulation hooks when available.

### Merge / Object Chain Puzzle

Applies when gameplay depends on object chains, board space pressure, spawn
tables, order goals, and long-tail item dependencies.

Typical generated specialized skills:
- merge-chain closure validation;
- spawn probability and bottleneck QA;
- board-space pressure simulation;
- order/quest feasibility QA;
- rare-item pacing and monetization pressure validation.

Required project inputs usually include item chain definitions, spawn tables,
board size, order goals, timers or energy rules, and economy constraints.

### Word / Dictionary Puzzle

Applies when answer validity, dictionary coverage, localization, ambiguity, or
unique-solution constraints determine quality.

Typical generated specialized skills:
- dictionary/license and locale coverage QA;
- unique-solution validation;
- clue-answer ambiguity QA;
- profanity/safety filtering;
- difficulty grading by word frequency, length, crossings, or clue opacity.

Required project inputs usually include dictionary source, locale policy,
answer list, clue list, board/grid constraints, safety rules, and validation
scripts when available.

## Hard Boundary

If a specialization proves generally reusable across many unrelated game types,
promote it later into a global bundled sub-skill. Until then, keep it
project-local so the global skill pack remains maintainable.
