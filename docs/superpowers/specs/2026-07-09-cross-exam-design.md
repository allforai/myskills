# cross-exam — 实证完成度盘问技能（设计）

日期：2026-07-09 · 状态：已与用户逐段确认 · 版本目标：claude/cross-exam v0.1.0

## 1. 问题与目标

大目标流水线（如 megastorm）跑完后经常"漏东漏西"：任务全绿，但交付物存在
**需求覆盖缺口**（某需求无任务覆盖）、**集成缝隙**（单任务验收全过、端到端串不起来）、
**需求跑偏**（做出来的不是想要的）、**细节质量马虎**（边角状态、错误处理粗糙）。
现有验收全部是**任务级、正向**的（supervisor 重跑 acceptance_cmd），回答"任务 T 过没过"，
回答不了"交付整体是不是用户要的东西"。

cross-exam 反向解决：以**泄漏点**（"如果 X 真做完了就不该长这样的细节"）为线头，
苏格拉底式引导用户对交付物提问，由技能**自主实测并截图取证**，最终产出一份
**实证完成度报告**。探针是手段，完成度报告是目的。

**非目标**：不修复任何缺口（只记账不修）；不在无人值守时运行（任务没全完不盘）；
不做全库预扫地图（被否的重型路线）。

## 2. 形态与边界（用户已冻结的决策）

- **独立通用技能** `/cross-exam`，新插件 `claude/cross-exam/`（CC 平台，暂不移植 Codex/OpenCode）。
  技能本体**零项目痕迹、零技术栈硬编码**；megastorm 只是可选数据源之一。
- **只在有人在场时运行**：run 全部收尾后（megastorm Phase 2 报告末尾一句邀请，不自动进入）
  或用户随时主动发起。执行中绝不插盘。
- **交互模式：苏格拉底式提问教练**——技能根据泄漏点生成"你该问这个系统的问题"，
  用户选题（或自己出题），技能实测作答，缺口在问答中浮现。
- **取证介质**：真实运行时 + 静态产物 + 流水线台账（若有）。**由 LLM 按问题自选介质**——
  有的问题读代码即可实证，有的必须跑起来逐状态截图，有的对台账对账即可。
- **证据纪律**：涉及运行时的问题，**逐状态截图留档**（不是只留成功图）。
- **下游**：只记账不修。缺口带证据入台账，修复由用户另行决定。
- **megastorm 邀请句在本计划范围内**：megastorm Phase 2 报告末尾加一句
  "可运行 /cross-exam 对本次交付做实证盘问"——这是对 megastorm 插件的附带小改
  （skill 一行 + 三处 manifest patch bump），排进实现计划，避免漏排或越界争议。

## 3. 架构 — 盘问官 / 实测官硬分离

```
claude/cross-exam/
├── .claude-plugin/{plugin.json, marketplace.json}   # v0.1.0
├── commands/cross-exam.md        # /cross-exam 斜杠入口
├── skills/cross-exam.md          # 盘问官主逻辑（会话侧）
├── knowledge/
│   ├── prompts/prober.md         # 实测官 prompt（fresh-context 子 agent）
│   ├── schemas.md                # ledger 条目 + 报告结构
│   └── lenses.md                 # 四镜头泄漏点扫描指南
└── scripts/render_report.py      # ledger.json → 完成度报告（确定性渲染）
```

- **盘问官**（主会话）：扫泄漏点、出问题牌、对话、解读证据、下裁决、记台账。持有全部上下文与怀疑。
- **实测官**（每问一个 fresh-context 子 agent）：只拿到问题 + 访问方式 + 证据落盘位置，
  **看不到盘问官的怀疑与对话史**。只带回原始观察，不下结论。

原理与 megastorm supervisor 同源反用：**测量者不能被期望污染**。
（正对头号教训：验收诚实性 > 引擎机制。）

产物落被测项目：`docs/cross-exam/<日期>-<目标>/{ledger.json, evidence/, completion-report.md}`。

## 4. 会话流程

1. **定靶（intake）**：确认被测对象、访问方式、需求基准
   （自动探测顺序：megastorm overview registry（R-*）→ specs → README → 问用户 → 无基准模式）。
   **无基准模式定义**：完全没有需求基准时，需求覆盖、需求跑偏两个镜头**关闭并在报告中
   明示"无基准，未盘"**，只开集成缝隙 + 细节质量两个镜头（它们不依赖基准）；报告
   `baseline: none`。环境能力探测（能否运行、有无 Playwright）。
   **安全确认：实测靶必须是本地/开发实例**（盘问会造真实调用），起手确认后才放开。
   **续跑**：intake 检测到未收敛 run 目录（判据：`ledger.json` 存在且
   `completion-report.md` 不存在）时，询问续接还是新开；
   run 目录命名 `docs/cross-exam/<日期>-<目标slug>/`。
2. **定面（facet map）**：从需求基准 + 代码拓扑摆出交付的面，用户勾选盘哪些、先盘哪个。
3. **盘问循环**（每轮一个面）：
   扫泄漏点 → 出 3 张问题牌 → 用户选一张（或自己出题）→ 派实测官 → 带回证据
   → 盘问官裁决 → 入台账 → 下一轮（从已发现缺口继续发散）。
4. **收敛**：用户喊停或面盘完 → `render_report.py` 从 ledger 确定性生成报告
   （脚本渲染而非口述，防上下文膨胀导致漏记/美化）。

## 5. 四镜头与问题牌规则

| 镜头 | 采样口 | 泄漏点示例 |
|------|--------|-----------|
| 需求覆盖 | 需求基准 ↔ 台账/代码对账 | R-07 说退款可追踪，全库无 refund_id |
| 集成缝隙 | 接口两侧形状对比 | 生产者 snake_case、消费者 camelCase；配置定义无人读 |
| 需求跑偏 | 需求原文 ↔ 实现语义 | 需求"离线可用"，实现"断网弹提示" |
| 细节质量 | 状态空间穷举 | 只有 happy path 测试名；错误分支裸 500 |

扫描浅而快，只找线头，不下结论。问题牌硬约束：

1. **每张牌挂一个泄漏点**（牌面三要素：问题、触发泄漏点、照亮哪个面）；
2. **必须可实测**（实测官能翻译成具体动作）；
3. **一轮 3 张覆盖不同泄漏点**（不许 3 张戳同一疑点推销怀疑）;
4. **涉及 UI/流程的牌注明状态清单**（加载/空/错误/成功…），实测官逐状态截图。

## 6. 实测官合同

输入（仅此，多一字不给）：

```json
{"question": "...", "target": {"how_to_run": "...", "entry": "...", "type": "web|cli|api"},
 "states_to_capture": ["..."], "evidence_dir": "docs/cross-exam/<run>/evidence/qNN/",
 "context_paths": ["<可选：台账/spec 等只读对账材料的路径，仅当该问需要>"]}
```

`context_paths` 只传路径不传盘问官的解读——台账对账类问题靠它拿到材料位置，
期望隔离原则不破。

纪律：

1. 自选介质并翻译为动作（读代码 / 起服务调接口 / Playwright 走 UI / 台账对账）——用户不动手；
2. 运行时取证逐状态截图，文件名带状态语义（`q07-02-waiting-35s.png`）；CLI/API 留原始输出文件；
3. 只观察不修：代码零改动，副作用限 intake 确认过的本地靶；
4. 返回原始观察不下结论：`{steps_taken, observations, exit_codes, output_excerpts, screenshots, could_not}`；
5. 测不了如实返回 `could_not` + 原因；
6. **每问必落证据目录（无一例外）**：运行时介质落截图/输出文件；读代码介质落
   摘录文件（路径 + 行号 + 原文引用）；台账介质落对账摘录；`could_not` 落原因文件
   （尝试了什么、卡在哪）。这保证 §7 红线（无证据拒渲）与"无法自证"裁决自洽——
   无法自证的证据就是"为什么无法自证"的记录本身；
7. **工具面收窄（机制而非嘱咐）**：prober.md 明示实测官禁止对项目源码使用
   Edit/Write，唯一可写路径是 `evidence_dir`。

裁决（盘问官，四种）：`实证完成` / `缺口` / `跑偏` / `无法自证`（需真设备/人工，如实挂账不猜）。

## 7. 台账与报告

**ledger.json**（逐问实时落盘，中断不丢）：target、baseline、facets（含
`examined|partial|not_examined`）、entries（q、facet、leak_point、medium、
evidence{dir, screenshots, key_observation}、verdict、requirement_ref、severity）。
`severity` 枚举 `high|medium|low`，由盘问官在裁决时判定（依据：需求引用的分量 +
实测后果的破坏面）；仅 `缺口`/`跑偏` 带 severity——`实证完成` 与 `无法自证` 不带
（后者不是缺口，是待人工验证项）。

**completion-report.md**（确定性渲染）：

1. 总览：N 面盘问 M 面、K 问、四类裁决计数；
2. 逐面完成度："该面 X 问中 Y 问实证通过"，链证据文件——**不编总百分比**；
3. 缺口清单：按严重度排，带证据与需求引用，可直接转修复任务；
4. 未盘问声明：未盘的面明示列出，报告只为盘过的部分作证。

**诚实性红线（写死在渲染脚本）**：`not_examined` 的面不进完成度统计；
无 evidence 目录的 entry 拒绝渲染（防口头裁决入报告）——因 §6.6 保证每问必落证据
（含 `could_not` 的原因文件），被拒渲的只可能是违规的口头裁决，四类裁决计数不受影响。

## 8. 错误处理

- 靶子起不来 → 本身入账为集成缝隙缺口，盘问降级为代码/台账介质继续；
- 实测官死/超时 → 重派一次（基础设施失败不算数），再失败记 `could_not`，绝不编造；
- 无 Playwright → UI 类问题裁决"无法自证"并注明环境缺口，其余介质照常。

## 9. 测试策略

- `render_report.py` 单元测试，含红线用例（not_examined 不计入、无证据拒渲）；
- 全流程 dogfood：真实小项目故意留 2-3 个已知缺口，跑一次 `/cross-exam`，
  验证缺口被晃出且截图落档（按项目测试纪律：不许只做 syntax check）。

## 10. 决策记录

| 决策 | 结论 | 备注 |
|------|------|------|
| 抓哪类漏 | 四类全要 | 覆盖缺口/集成缝/跑偏/细节 |
| 运行时机 | 收尾后 + 随时，仅交互式 | 无人值守不盘；执行中不盘 |
| 交互模式 | 苏格拉底式提问教练 | 用户在提问中自己发现缺口 |
| 取证介质 | 运行时+静态+台账，LLM 自选 | git 历史不要 |
| 证据 | 逐状态截图留档 | 不只留成功图 |
| 下游 | 只记账不修 | 审计定位，无修复动机 |
| 落点 | 独立通用技能，CC only | megastorm 仅为数据源 |
| 命名 | /cross-exam 盘问 | 探针是手段不是目的 |
| 产出 | 实证完成度报告 | 报告是目的 |
| 架构 | 盘问官/实测官分离（路线 B） | 测量不被期望污染 |
