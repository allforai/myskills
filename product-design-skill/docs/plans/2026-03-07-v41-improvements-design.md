## #1 Implementation Contract

### Problem

`experience-map.json` specifies screen patterns (overlay, bottom-sheet, portal) via `non_negotiable`, but task-execute may simplify them to plain page routes. Design intent gets lost at the code boundary.

### Solution

Add `implementation_contract` field to each screen in `gen_experience_map.py`:

```json
{
  "id": "S003",
  "implementation_contract": {
    "pattern": "bottom-sheet",
    "forbidden": ["page-route", "full-screen-modal"],
    "required_behaviors": ["swipe-to-dismiss", "backdrop-tap-close"]
  }
}
```

**Derivation rules** (in `build_screens_for_node()`):
- `pattern`: inferred from node's `ux_intent` + screen CRUD type
  - intent contains "quick" / "overlay" / "confirm" → `bottom-sheet`
  - intent contains "detail" / "full" → `full-page`
  - intent contains "select" / "pick" → `modal-picker`
  - CRUD "C" with high emotion_intensity → `multi-step-form`
  - default → `standard-page`
- `forbidden`: inverse of pattern (bottom-sheet forbids page-route)
- `required_behaviors`: derived from pattern preset map

**Verification** (in `product-verify.md`):
- Static phase: grep code for screen component, check pattern markers
- If violation found → generate FIX_CONTRACT task type

### Files

- `gen_experience_map.py`: add contract derivation in `build_screens_for_node()`
- `experience-map-schema.md`: document `implementation_contract` field
- `experience-map.md`: mention contract in output description
- `product-verify.md`: add contract verification step in static phase

## #3 Micro-Interactions

### Problem

Design spec covers page-level components and states but not the micro-interaction layer (animation, easing, haptic feedback) that determines product feel.

### Solution

Add `micro_interactions` section to ui-design-spec output in `gen_ui_design.py`, auto-derived from `emotion_intensity` using presets:

```json
{
  "screen_id": "S003",
  "micro_interactions": [
    {
      "trigger": "primary-action-tap",
      "animation": "scale-bounce",
      "duration_ms": 300,
      "easing": "cubic-bezier(0.34, 1.56, 0.64, 1)",
      "haptic": "impact-medium",
      "emotion_alignment": "satisfying confirmation"
    }
  ]
}
```

**Preset tiers** (based on `emotion_intensity`):

| Intensity | Tier | Animation | Duration | Easing | Haptic |
|-----------|------|-----------|----------|--------|--------|
| 1-3 | calm | fade | 200ms | ease-out | none |
| 4-6 | moderate | slide + fade | 250ms | ease-in-out | light |
| 7-10 | intense | scale-bounce | 300ms | spring overshoot | impact-medium |

**Additional rules:**
- Success states (CRUD "C" completion) always get `scale-bounce` + `impact-medium` regardless of intensity
- Error states always get `shake` + `notification-error`
- Loading states get `pulse` animation

**Output:** Appended as `## Micro-Interaction Specifications` section in `ui-design-spec.md`, plus `micro-interactions.json` as machine-readable output.

### Files

- `gen_ui_design.py`: add preset mapping + spec section + JSON output
- `ui-design.md`: document micro-interactions output

## #4 Token Single Source of Truth

### Problem

Design tokens (colors, spacing, radius, typography) defined in spec doc, manually copied to config files by task-execute. Two copies drift independently.

### Solution

Export `tokens.json` from `gen_ui_design.py` alongside `ui-design-spec.md`:

```json
{
  "color": {
    "primary": "#6750A4",
    "on_primary": "#FFFFFF",
    "secondary": "#625B71",
    "tertiary": "#7D5260",
    "background": "#FFFBFE",
    "surface": "#FFFBFE",
    "surface_variant": "#E7E0EC",
    "on_surface": "#1C1B1F",
    "on_surface_variant": "#49454F",
    "error": "#B3261E",
    "success": "#2E7D32",
    "warning": "#ED6C02"
  },
  "spacing": {
    "unit": 4,
    "scale": [4, 8, 12, 16, 24, 32]
  },
  "radius": {
    "sm": 4,
    "md": 12,
    "lg": 16
  },
  "typography": {
    "display": {"size": 57, "weight": 400},
    "headline": {"size": 32, "weight": 400},
    "title": {"size": 22, "weight": 500},
    "body": {"size": 16, "weight": 400, "line_height": 24},
    "label": {"size": 14, "weight": 500}
  },
  "font": "'Roboto', 'Noto Sans SC', sans-serif",
  "shadow": "0 1px 3px 1px rgba(0,0,0,.15), 0 1px 2px rgba(0,0,0,.3)",
  "style_name": "Material Design 3",
  "generated_at": "2026-03-07T12:00:00"
}
```

**Source:** The existing `STYLES` dict in `gen_ui_design.py` already has all these values. We extract and write to `tokens.json`.

**Downstream consumption** (documented in `project-scaffold.md`):
- Scaffold reads `tokens.json` if present
- Generates framework-specific config (Tailwind theme / CSS variables / Flutter theme)
- Token changes propagate by re-running ui-design + scaffold

### Files

- `gen_ui_design.py`: write `tokens.json` from STYLES dict
- `ui-design.md`: document tokens.json output
- `project-scaffold.md`: document tokens consumption

## #5 LLM Cognitive Walkthrough

### Problem

E2E tests verify features work but not whether users can discover them. A button may exist and function correctly yet be invisible to real users.

### Solution

Add LLM-as-user walkthrough step in `product-verify.md` dynamic phase:

1. Set LLM persona from role-profiles (e.g., "new user wanting to place first order")
2. Do NOT provide route names, component IDs, or navigation hints
3. LLM explores via Playwright snapshots autonomously
4. Record: click count to complete task, stuck points, confusion nodes
5. Map findings to experience-map nodes

**Output:** `cognitive-walkthrough.json` with per-task metrics:

```json
{
  "walkthroughs": [
    {
      "persona": "R01 first-time buyer",
      "goal": "place first order",
      "result": "completed",
      "click_count": 7,
      "expected_clicks": 5,
      "stuck_points": [
        {
          "at_screen": "S003",
          "node": "N0103",
          "reason": "add-to-cart button below fold",
          "recovery": "scrolled down after 2 failed attempts"
        }
      ],
      "discoverability_score": 0.71
    }
  ],
  "overall_discoverability": 0.85
}
```

**Trigger:** Only when `dynamic` or `full` mode, AND Playwright is available. Skip silently otherwise.

### Files

- `product-verify.md`: add walkthrough step after existing dynamic tests

## #6 Stitch Decision Point

### Problem

When Stitch is unavailable, the pipeline silently falls back. Users may not realize visual quality was never verified.

### Solution

Before Phase 5-7 parallel execution in `SKILL.md` and `commands/product-design.md`, add:

```
If Stitch is unavailable:
  AskUserQuestion with 3 options:
  A) Upload design mockups manually
  B) Skip visual acceptance (with explicit acknowledgment)
  C) Configure Stitch and continue

If B chosen:
  - Continue without Stitch
  - forge-report prominently marks "Visual quality NOT verified"
  - design-audit flags "stitch_skipped: true"
```

No silent degradation. The user must consciously acknowledge the quality tradeoff.

### Files

- `SKILL.md`: add Stitch check before parallel phase
- `commands/product-design.md`: add Stitch check in full orchestration
