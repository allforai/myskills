# 防御性模式参考

> 通用防御模式定义。各技能文件的「防御性规范」章节引用本文档，并补充技能特有的应用细节。

---

## 模式 A：JSON 加载校验

**触发条件**：加载任何 `.allforai/` 下的 `.json` 产物时。

**处理流程**：

1. 读取文件内容
2. 用 `python -m json.tool` 或等效方式验证 JSON 合法性
3. 解析失败 →
   - 检查同目录是否有 `.bak` 备份文件
   - 有 `.bak` → 提示用户：「⚠ {文件名} JSON 解析失败，发现备份文件 {文件名}.bak，是否恢复？」
   - 无 `.bak` → 提示用户：「⚠ {文件名} JSON 解析失败，请重新执行 {对应技能} 重新生成」
4. **不静默跳过，必须告知用户**

**示例**：

```
⚠ task-inventory.json 解析失败（第 42 行语法错误）
  发现备份: task-inventory.json.bak (2h 前)
  选项: (a) 恢复备份 (b) 重新执行 product-map
```

## 模式 B：零结果检测

**触发条件**：每个关键生成步骤完成后，检查产出数量。

**处理流程**：

1. 步骤完成后统计产出条目数
2. count > 0 → 正常继续
3. count == 0 → **区分两种情况**：
   - **确实为零**（检查逻辑正常完成，无异常，数据源充分）→ ✓ 明确告知：「检查完毕，{检查内容}确实为零/无问题」
   - **可能异常**（上游数据不足、加载失败、搜索无结果）→ ⚠ 警告：「产出为零，可能是 {原因}，建议 {操作}」

**关键原则**：

- 不输出空列表而不解释
- ✓ 标记表示"真的没有"，⚠ 标记表示"可能有问题"
- 向用户展示明确消息，而非静默的空数组

**示例**：

```
✓ 所有检查通过，未发现功能缺口（共检查 45 个任务）

⚠ 提取到 0 个任务。上游 task-inventory.json 仅含 2 个角色且均为 user_removed 状态，
   建议检查 product-map 是否需要更新。
```

## 模式 C：规模自适应

**触发条件**：技能开始执行时，根据处理对象数量判定规模等级。

**通用阈值**（各技能可微调，但模式结构一致）：

| 规模 | 判定条件 | 交互策略 |
|------|----------|----------|
| **small** | 对象数 ≤ 30 | 逐项展示，逐步确认 |
| **medium** | 对象数 31–80 | 摘要模式，按模块/角色分组确认 |
| **large** | 对象数 > 80 | 脚本生成输出文件 + 统计摘要 + 仅展开有问题项 |

**计量单位因技能而异**：

| 技能 | 计量对象 |
|------|----------|
| product-map | 任务数 |
| experience-map | 界面数 |
| use-case | 任务数（决定用例量） |
| feature-gap | 任务数 |
| ui-design | 界面数（阈值 ≤15 / 16–40 / >40） |
| design-audit | 任务数 |
| product-concept | 不适用（概念层不分规模） |

**展示规则**：

- **small**：完整列表 + 逐条确认
- **medium**：分组摘要 + 仅展开有 flag/问题的项
- **large**：统计总览 + 仅展开高优先级/有问题的项

## 模式 D：网络搜索不可用处理

**触发条件**：任何步骤调用网络搜索时。

**处理流程**：

1. 尝试 网络搜索
2. **区分两种失败**：

**情况 1：工具正常但无有用结果**
- 换关键词重试一轮（调整搜索语言或搜索角度）
- 仍无有用结果 → 告知用户：「该方向公开信息较少，已搜索 {N} 轮，未找到高质量结果」
- 继续流程，标注 `search_status: "no_useful_results"`

**情况 2：工具本身报错/不可用**
- 告知用户：「⚠ 网络搜索暂不可用（{错误信息}）」
- 提供选项（向用户确认）：
  - (a) 跳过搜索，继续流程（使用 AI 已有知识 + 用户输入）
  - (b) 用户手动提供参考资料/URL
- 标注 `search_status: "tool_unavailable"`

**关键原则**：

- 不静默跳过搜索步骤
- 必须告知用户搜索状态
- 区分"搜不到"和"工具坏了"

## 模式 E：上游数据过期检测

**触发条件**：加载上游产物的 JSON 文件时。

**处理流程**：

1. 读取上游产物的 `generated_at` 字段
2. 读取本技能上次运行的决策文件（`*-decisions.json`）的最新 `decided_at` 时间戳
3. **比较逻辑**：
   - 上游 `generated_at` **晚于**本技能最近 `decided_at` → ⚠ 警告：「上游 {文件名} 在本技能上次运行后被更新（{上游时间} > {本技能时间}），当前结果可能基于旧版上游数据，建议重新运行本技能」
   - 上游 `generated_at` **早于或等于**本技能最近 `decided_at` → 正常，无警告
4. **仅警告，不阻断流程**

**示例**：

```
⚠ 上游数据已更新:
  task-inventory.json generated_at: 2026-02-27T14:00:00Z
  本技能上次运行:                   2026-02-25T10:00:00Z
  建议: 重新执行 use-case 以获取最新数据
```

## 模式 F：引用完整性断言

**触发条件**：当步骤 N 引用步骤 M 产出中的 ID（如 task_id、screen_id、flow_id）时。

**处理流程**：

1. 收集当前步骤中所有引用的外部 ID
2. 逐一检查该 ID 是否存在于对应的上游产出中
3. **断言失败** → 标记 `BROKEN_REF`，记录：
   - 引用源（当前步骤 + 产物项）
   - 被引用 ID
   - 预期存在的上游产出文件
4. **不中断流程**，但记录到报告中
5. 向用户展示所有 `BROKEN_REF`

**示例**：

```
⚠ 引用完整性检查: 发现 2 个断裂引用
  - Step 3 旅程 J005 引用 task_id T999 → task-inventory 中不存在
  - Step 2 用例 UC042 引用 screen_ref S088 → experience-map 中不存在
  流程继续，但建议修复上游数据。
```

## 模式 G：JSON .bak 自动恢复

**触发条件**：写入任何 `.allforai/` 下的关键 JSON 产物时。

**适用文件**：所有 `*-decisions.json`、`*-report.json`、`*-profile.json`、`*-plan.json` 等关键产物。

**处理流程**：

1. **写入前**：若目标文件已存在，复制为 `{filename}.bak`
2. **写入完成**：保留 `.bak`（直到下次成功写入时覆盖）
3. **加载时**：JSON 解析失败 → 尝试读取 `.bak` → 成功则恢复并警告「已从备份恢复」→ `.bak` 也失败 → 报错终止

**与模式 A 的关系**：

- 模式 A（JSON 加载校验）在解析失败时检查 `.bak` 是否存在并提示用户
- 模式 G 确保 `.bak` 文件始终存在（每次成功写入前自动备份）
- 两者配合形成完整的写入保护 + 加载恢复链路

**示例**：

```
写入 gap-tasks.json:
  1. 检测到已有 gap-tasks.json → 复制为 gap-tasks.json.bak
  2. 写入新内容到 gap-tasks.json
  3. 写入成功 → 保留 .bak 直到下次写入

加载 gap-tasks.json:
  1. JSON.parse 失败 → 尝试 gap-tasks.json.bak
  2. .bak 解析成功 → 恢复到主文件 + 警告「⚠ gap-tasks.json 已从备份恢复，请检查上次写入是否中断」
  3. .bak 也失败 → 报错终止「⚠ gap-tasks.json 及其备份均无法解析，请重新执行 feature-gap」
```

## 模式 H：用户中断部分保存（原则）

**适用场景**：用户在多步骤流程中中途中断（如关闭对话、切换任务）。

**原则**：

- 每个 Step 确认后立即写入磁盘（已有设计，各技能的 decisions.json 机制）
- 未确认的 Step 产出仅存在于对话上下文中，不落盘
- 下次运行时从最后一个已确认 Step 恢复
- 中断时已生成但未确认的内容不保证保留

> 本模式仅在此定义原则，各技能已有的 decisions.json 增量复用机制天然支持，无需额外实现。
