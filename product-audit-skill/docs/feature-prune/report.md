# Step 4: 剪枝报告生成

> 本步骤汇总前三步结果，生成最终剪枝报告和可操作行动清单。

---

## 1. 报告生成流程

报告生成严格按以下顺序执行：

1. **汇总前序结果** — 聚合 Step 0（功能清单）、Step 1（核心场景识别）、Step 2（必要性评估）、Step 3（竞品参考，如有）的全部数据
2. **生成 3 个输出文件**：
   - `prune-report.md` — 完整剪枝报告（人类可读）
   - `prune-actions.json` — 行动清单（CUT 如何处理、DEFER 推荐时机）
   - `prune-decisions.json` — 用户决策日志（跨次剪枝持久化）
3. **在对话中输出摘要** — 这是**强制行为**，不可省略，不可仅生成文件而不展示

三个文件写入项目根目录下的 `.allforai/feature-prune/` 目录。

---

```markdown
- **复杂度比**: {high/medium/low} — 涉及 {X} 个文件
- **竞品覆盖**: {X/N 竞品包含此功能}（仅 full/scope 模式）
- **建议行动**: 从需求文档移除或归档到 backlog

## DEFER 功能详情

逐项说明：

### F-XXX: {功能名称}

- **来源**: {file}:{line}
- **关联场景**: {S-XXX — 关联描述}
- **推荐实现时机**: {如: 核心场景稳定后 / v2.0 阶段}
- **竞品覆盖**: {X/N 竞品包含此功能}（仅 full/scope 模式）

## 竞品参考摘要（仅 full/scope 模式）

| 功能 | 本项目 | 竞品A | 竞品B | 竞品C | 覆盖率 |

|------|------|----------|----------|------|
| {time} | F-XXX {name} | CUT | DEFER | {用户给出的原因} |
```

### 模板使用规则

- `{timestamp}` — ISO 8601 格式，如 `2026-02-24T10:30:00Z`
- `{mode}` — `full` / `quick` / `scope`
- 占比计算基于功能总数（CORE + DEFER + CUT）
- 竞品参考摘要仅在 `full` 或 `scope` 模式下生成（`quick` 跳过 Step 3，无竞品数据）
- 所有 "详情" 部分，如果该分类数量为 0，输出 "无" 而不是省略该节

## 3. prune-actions.json 格式

行动清单用于后续消费，每个 CUT 或 DEFER 功能对应一个 action：

```json
{
  "prune_timestamp": "2026-02-24T10:30:00Z",
  "prune_mode": "full",
  "actions": [
    {
      "id": "A-001",
      "type": "CUT",
      "feature_id": "F-015",
      "feature_name": "数据可视化大屏",
      "action": "从需求文档移除或归档到 backlog",
      "source": "docs/prd.md:120",
      "status": "open"
    },
    {
      "id": "A-002",
      "type": "DEFER",
      "feature_id": "F-008",
      "feature_name": "数据归档",
      "action": "标注推荐实现时机: 核心场景稳定后",
      "recommended_timing": "核心场景稳定后",
      "source": "docs/prd.md:95",
      "status": "open"
    }
  ]
}
```

### Action ID 规则

- `A-001` 起始，按分类顺序递增（先 CUT，后 DEFER）
- 每个 CUT / DEFER 功能对应一个 action

### Action 字段说明

| 字段 | CUT 类型 | DEFER 类型 |
|------|----------|------------|
| `id` | 必填 | 必填 |
| `type` | `"CUT"` | `"DEFER"` |
| `feature_id` | 必填 | 必填 |
| `feature_name` | 必填 | 必填 |
| `action` | "从需求文档移除或归档到 backlog" | "标注推荐实现时机: {时机}" |
| `recommended_timing` | 不适用 | 必填（如 "核心场景稳定后"） |
| `source` | 必填 — 需求源 `file:line` | 必填 — 需求源 `file:line` |
| `status` | `open` | `open` |

### Status 值

- `open` — 新标记，待处理
- `resolved` — 已按建议处理（后续运行自动更新）
- `deferred` — 用户决策为延期（DEFER 类型默认）
- `dismissed` — 用户决策为忽略此建议

## 4. prune-decisions.json 格式

决策日志是**跨剪枝持久化**的核心文件：

```json
{
  "version": "1.0.0",
  "last_updated": "2026-02-24T10:30:00Z",
  "decisions": [
    {
      "feature_id": "F-015",
      "feature_name": "数据可视化大屏",
      "suggested": "CUT",
      "final": "DEFER",
      "reason": "客户明确要求过，先延后观察",
      "decided_at": "2026-02-24T10:15:00Z"
    },
    {
      "feature_id": "F-013",
      "feature_name": "订单归档",
      "suggested": "DEFER",
      "final": "DEFER",
      "reason": "确认延后，核心场景稳定后实现",
      "decided_at": "2026-02-24T10:20:00Z"
    }
  ]
}
```

### 持久化规则

- **APPEND-ONLY** — 单次剪枝会话中只追加，不修改已有条目
- **跨次剪枝保留** — 下次运行 `scope` 模式时自动读取并应用
- **同一功能多次决策** — 保留所有记录，以最新的 `decided_at` 为准
- **决策字段说明**:
  - `suggested` — Claude 的初步分类建议（CORE / DEFER / CUT）
  - `final` — 用户的最终决定（CORE / DEFER / CUT）
  - `reason` — 用户给出的理由（如用户未说明，记录 "用户确认"）

### 自动应用逻辑

在 `scope` 模式下读取此文件时：

1. 对于每个已有决策的功能，展示历史决策，询问是否沿用（而非直接跳过）
2. 在报告中标注 "（沿用上次决策）" 或 "（本次重新评估）"
3. 仅对新发现的或无历史决策的功能执行完整评估流程

### 决策复用与失效

| 决策类型 | 复用条件 | 失效条件 |
|----------|----------|----------|
| 分类决策（CORE/DEFER/CUT） | 无条件复用 | 用户手动修改 |
| 场景归属覆写 | 无条件复用 | 用户手动取消 |
| 竞品信号覆写 | 无条件复用 | 用户手动取消 |
| 评估维度确认 | 对应代码未变动时复用 | 对应代码有 git diff 变动 |

## 5. 对话输出要求

生成文件后，**必须**在对话中输出报告摘要。这是强制行为，不可省略。

### 必须包含的内容

**剪枝总览表格** — 与 prune-report.md 中的剪枝总览相同：

```
剪枝总览:
| 分类 | 数量 | 占比 |
|------|------|------|
| CORE | 12 | 60% |
| DEFER | 5 | 25% |
| CUT | 3 | 15% |
```

**核心场景列表** — 列出每个场景及其 CORE 功能：

```
核心场景:
- S-001 商家处理客户订单: F-010 订单列表, F-011 订单详情, F-012 订单导出
- S-002 系统管理: F-001 用户管理, F-002 权限配置
```

**每一个 CUT 功能** — 必须列出，附来源引用和理由：

```
CUT 功能:
- F-015 数据可视化大屏 — 来源: docs/prd.md:120 — 无明确场景归属，涉及 12 个文件
- F-019 实时聊天 — 来源: docs/prd.md:180 — 无明确场景归属，无其他功能依赖
```

**每一个 DEFER 功能** — 必须列出，附推荐时机：

```
DEFER 功能:
- F-008 数据归档 — 来源: docs/prd.md:95 — 推荐时机: 核心场景稳定后
- F-013 订单归档 — 来源: docs/prd.md:110 — 推荐时机: 核心场景稳定后
```

**孤立功能列表** — 无场景归属的功能必须单独列出：

```
孤立功能（无场景归属）:
- F-015 数据可视化大屏 — CUT — 无明确场景归属
- F-019 实时聊天 — CUT — 无明确场景归属
```

**竞品参考摘要**（仅 full/scope 模式）— 覆盖率矩阵：

```
竞品参考:
| 功能 | 本项目 | 竞品A | 竞品B | 竞品C | 覆盖率 |
|------|--------|-------|-------|-------|--------|
| 数据可视化大屏 | CUT | 无 | 无 | 无 | 0/3 |
| 数据归档 | DEFER | 有 | 有 | 无 | 2/3 |
```

**用户决策日志摘要** — 本次会话中所有与建议不一致的决策：

```
用户决策（与建议不同）:
- F-015 数据可视化大屏: 建议 CUT → 用户决定 DEFER — 原因: 客户明确要求过
```

## 6. 禁止省略

- 不可用 "等 X 个功能" 省略任何条目
- 不可用 "详见报告文件" 替代对话输出
- 不可用 "详见 prune-report.md" 跳过对话中的展示
- 每个分类如果数量为 0，输出 "无" 而不是省略该节
- CUT 功能必须逐条列出，即使有 20 个也必须全部展示
- DEFER 功能必须逐条列出，附推荐实现时机
- 孤立功能必须逐条列出，附分类和理由
- 用户决策必须逐条列出与建议不一致的项

## 7. 词汇纪律检查

在输出报告前，执行自检。报告中的每一句话必须是**事实陈述**，基于场景分析和三维评估数据。

### 禁用词汇（出现即违规）

| 禁用表达 | 应替换为 |
|----------|----------|
| 此功能不应该存在 | 此功能无明确场景归属 |
| 你不需要这个功能 | 此功能在已识别的核心场景中无关联 |
| 过度设计 | 实现涉及 {X} 个文件，复杂度比为 high |
| 必须延后 | 建议分类为 DEFER，推荐时机: {描述} |
| 建议添加XX功能 | （删除，不输出 — 违反"只剪不加"铁律） |
| 需要补充XX | （删除，不输出 — 剪枝不做添加建议） |
| 最佳实践 | （删除，不输出） |
| 可以考虑 | （删除，不输出） |
| 推荐使用 | （删除，不输出） |

### 合规表达示例

合规：
- "F-015（数据可视化大屏）无明确场景归属，实现涉及 echarts + WebSocket 共 12 个文件，建议分类为 CUT — 来源: docs/prd.md:120"
- "F-013（订单归档）与'商家处理客户订单'场景关联度为中等，复杂度涉及归档策略 + 定时任务共 6 个文件，建议分类为 DEFER — 来源: docs/prd.md:110"
- "竞品在同阶段普遍不包含'数据可视化大屏'功能（0/4 覆盖）"
- "竞品 3/4 包含'甘特图'功能，本项目暂未规划。仅标注，不纳入建议范围。"

违规：
- "数据可视化大屏过度设计了，不应该存在"
- "你不需要数据可视化大屏，竞品都没做"
- "建议添加甘特图功能，竞品都有"
- "必须把订单归档延后到 v2.0"

### 引用要求

每一条发现必须附带 `file:line` 引用：

- CUT — 引用需求源位置（如 `docs/prd.md:120`）或代码位置
- DEFER — 引用需求源位置（如 `docs/prd.md:95`）或代码位置
- CORE — 引用需求源位置或代码位置

无法提供引用的发现不应出现在报告中。

## 8. 输出文件写入顺序

1. 先写 `prune-report.md`（完整剪枝报告）
2. 再写 `prune-actions.json`（行动清单）
3. 再写/追加 `prune-decisions.json`（决策日志）
4. 最后在对话中输出摘要

如果 `prune-decisions.json` 已存在（如 scope 模式下），读取后追加新决策，不覆盖已有内容。写入合并逻辑：

1. 读取已有 `prune-decisions.json`
2. 将本次会话的新决策追加到 `decisions` 数组
3. 更新 `last_updated` 时间戳
4. 写回文件
