# Execution Engine Protocol: Subagent Isolation + Upstream Feedback Loop

## Problem

LLM agent 在执行多阶段 skill 时有两个结构性问题：

1. **注意力衰减**：主流程 context 随执行推进不断膨胀（规则、源码、中间产物、错误日志），每条规则被遵守的概率随之下降。不是 LLM 不知道规则，是注意力被稀释了。

2. **单向流水线丢信息**：上游阶段的遗漏在下游才暴露，但下游只能在自己的层面打补丁，不能回退到上游修根因。结果是"漏东漏西"贯穿全链路。

## Design Principles

- **质量优先于速度**：宁可多耗 token，不可降低执行质量
- **协议要薄，LLM 判断空间要大**：不硬编码业务逻辑、不预定义 skill 依赖链，让 LLM 根据证据自行推理
- **下游是上游最好的质检员**：每个阶段都有权（也有义务）检查上游产出，发现问题时回退到根因
- **不特化**：协议与具体 skill/业务领域无关，任何 skill 直接套用

---

## Layer 1: 主流程角色——纯调度器

主流程**不执行任何业务逻辑**，只做三件事：

1. **读状态**：当前在哪个阶段、上一个 subagent 返回了什么
2. **决策路由**：正常 → 下一阶段；`UPSTREAM_DEFECT` → 回退到目标阶段
3. **Dispatch subagent**：组装任务描述（阶段输入 + 该阶段需要的规则文件路径） → 发出去

### 主流程 context 只包含

- 阶段依赖图（几十行）
- 当前进度状态（几行）
- 上一个 subagent 的结果摘要（几行）

### 不在主流程 context 里的

- 铁律全文（由 subagent 自己加载）
- 源代码内容（由 subagent 自己读）
- 中间产物细节（写入 `.allforai/`，subagent 自己读）

主流程 context 始终很短，不会随执行推进而膨胀。

---

## Layer 2: Subagent 间信息共享

隔离的核心矛盾：**隔离太彻底 → 信息孤岛，隔离不够 → 注意力污染**。

解法：subagent 之间不直接通信，通过两层机制共享信息：

### 2.1 产物文件（结构化数据传递）

每个阶段的 subagent 将产出写入 `.allforai/` 目录下的文件（JSON 为机器读，Markdown 为人读）。下游 subagent 直接读取上游产物文件作为输入。

- **写**：该阶段 subagent
- **读**：下游所有需要的 subagent
- **特点**：完整、结构化、持久化，是信息共享的主通道

### 2.2 阶段摘要（精炼上下文传递）

每个 subagent 完成时，除了写产物文件，还返回一段**精炼摘要（≤500 字）**给主流程。摘要包含：

- 做了什么（关键决策）
- 发现了什么（异常/风险）
- 下游需要注意什么

主流程**选择性地**把相关摘要注入下一个 subagent 的 prompt 里——不是把所有历史摘要都塞进去，而是由主流程（LLM）判断哪些前序摘要和当前阶段相关。

- **写**：subagent 返回时附带
- **读**：主流程 + 主流程选择性注入给下一个 subagent
- **特点**：轻量、高信噪比，是产物文件的补充而非替代

### 为什么需要两层

| | 产物文件 | 阶段摘要 |
|--|---------|---------|
| 信息量 | 完整细节 | 精炼结论 |
| 消耗 context | subagent 按需读取部分内容 | 主流程注入，几百字 |
| 适合传递 | 数据、清单、配置 | 决策理由、风险提示、注意事项 |

## Layer 3: 下游回退协议（Upstream Feedback）

### 3.1 UPSTREAM_DEFECT 信号

当 subagent 在执行中发现"这不是我能修的，是上游的问题"时，返回标准信号：

```json
{
  "signal": "UPSTREAM_DEFECT",
  "source_phase": "<当前阶段的 skill.phase>",
  "target_phase": "<应该修复的阶段的 skill.phase>",
  "defect_type": "<LLM 自行分类>",
  "evidence": "<具体证据，越详细越好>",
  "severity": "blocker | warning",
  "suggested_fix": "<建议修复方向>"
}
```

**关键设计点**：

- `target_phase` 由发现缺陷的 subagent 自己判断，不由协议预定义。LLM 根据证据推断该谁修
- `defect_type` 不限定枚举值，LLM 自行描述缺陷类型
- `suggested_fix` 是建议而非指令，上游 subagent 有自主判断权

### 3.2 主流程回退决策

```
severity = blocker → 立即回退，暂停当前阶段
severity = warning → 记录，继续当前阶段，当前阶段完成后再回退
```

### 3.3 回退执行流程

1. 主流程加载目标阶段所属 skill 的阶段声明
2. Dispatch 目标阶段的 subagent，prompt 里注入缺陷信号作为**修复任务**（不是从头重跑整个阶段）
3. 上游 subagent 修复 → 更新产物文件 → 返回修复摘要
4. 主流程把修复摘要注入当前阶段 subagent → 从断点继续（不是从头重跑）

### 3.4 防无限回退

- 同一个 `{source_phase, target_phase}` 对最多回退 2 次
- 第 3 次 → 标记为 `UNRESOLVED_DEFECT`，继续执行，最终报告中标红
- 这是协议级约束，与具体 skill 无关

### 3.5 跨 skill 回退

回退不区分 skill 内和跨 skill——协议完全一样。任何 skill 的任何阶段都可以向任何其他 skill 的任何阶段发出回退信号，只要有证据支撑。主流程按 `target_phase` 解析出目标 skill 和目标步骤，加载对应阶段声明，dispatch 修复 subagent。

## Layer 4: Subagent 任务描述模板

每个 subagent 被 dispatch 时，主流程用统一模板组装 prompt：

```
## 任务
{阶段名称}：{一句话任务描述}

## 输入
- 读取：{产物文件路径列表}
- 上游摘要：{主流程选择性注入的前序阶段摘要}
- 修复上下文：{如有 UPSTREAM_DEFECT 触发的修复任务，附在这里；正常流程无此字段}

## 规则
读取以下规则文件后执行：
- {该阶段需要的规则文件路径列表}

## 输出要求
1. 产物文件：写入 {输出路径}
2. 阶段摘要：返回 ≤500 字摘要（关键决策 + 发现 + 下游注意事项）
3. 缺陷信号：若发现上游问题，按 UPSTREAM_DEFECT 格式返回；无问题则不返回
```

**设计要点**：

- **规则按需加载**：每个 subagent 只读自己需要的规则文件，不加载全量
- **上游摘要是可选的**：主流程判断哪些前序摘要和当前阶段相关，只注入相关的
- **修复上下文只在回退时注入**：正常流程不带这个字段，subagent 看到它就知道自己是来修问题的

## Layer 5: Skill 声明式适配

每个 skill 不需要自己实现调度逻辑，只需要在 SKILL.md 或 execution-playbook 中声明阶段结构：

```yaml
phases:
  - id: <阶段ID>
    subagent_task: "<一句话任务描述>"
    input: [<产物文件路径列表>]
    output: "<输出产物路径>"
    rules: [<规则文件路径列表>]
    depends_on: [<前置阶段ID列表>]  # 可选，无依赖则可并行
```

**这不是新文件格式**——这段声明写在 skill 的入口文件里，主流程（LLM）直接理解 YAML。不需要 parser，不需要运行时框架。

**没有预定义的 skill 间依赖链**。回退路由完全由 subagent 在运行时根据证据推断。

**`depends_on` 只描述 skill 内部的阶段顺序**——没有依赖关系的阶段可以并行 dispatch（顺带解决方案 A 的效率问题）。

---

## Summary

| 层 | 职责 | 核心原则 |
|----|------|---------|
| 主流程 | 纯调度，context 极简 | 不执行业务逻辑 |
| 信息共享 | 产物文件 + 精炼摘要 | 隔离但不孤岛 |
| 回退协议 | UPSTREAM_DEFECT 信号 | 下游是上游的质检员 |
| 任务模板 | 统一 dispatch 格式 | 规则按需加载 |
| Skill 适配 | 声明式阶段图 | 协议通用，不特化 |
