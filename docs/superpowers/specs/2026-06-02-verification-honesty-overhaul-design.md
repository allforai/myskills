# Verification-Honesty Overhaul — Design Spec

**Date:** 2026-06-02
**Status:** Design → implementing
**Scope:** `claude/meta-skill/` — the verification/acceptance layer.
**Trigger:** meta-skill reported TeteChat at 94.65/100 "launch-ready"; real completeness was **< 30%** — a 3× systematic inflation.

---

## 1. Problem

The verification layer **lies about completeness** because it has **no ground truth**. Every green signal is written by the same system that generated the code:

```
generate node → self-reports "completed"
   → verify node reads the self-report + artifact files → writes a "PASS" report.json
      → acceptance reads the PASS reports → scores 94.65
```

It is PASS-reports all the way down; nothing ever runs the real product and observes whether the feature actually works. Three concrete failure modes:

1. **`completed` ≠ works.** A node passes if `exit_artifacts` exist and contain no literal `placeholder`/`stub` string. "Compiles + returns fake success + half-wired UI" slips straight through (exactly the behavior MEMORY 2026-05-29 had to ban by hand).
2. **Measures artifacts, not reality.** "Runtime smoke PASS / iOS 35 tests PASS" — reports written by agents that likely never started a service or hit an endpoint; tests generated by the same agents exercise mocks/stubs, not real behavior.
3. **Self-graded homework.** The verifier trusts (or *is*) the generator; nobody is required to *disprove* "done."

Same failure mode this very session surfaced: the run-engine shipped with placeholder agent prompts yet **41 green tests** (fakes keyed on labels hid the empty prompts). **Green ≠ real.**

---

## 2. Principle: epistemic honesty

> **"Generated" must never masquerade as "verified."**

Today a node is binary `completed`/`failed`, which collapses into a false PASS. The fix introduces a **third state** and an **evidence requirement**, so the completeness number cannot inflate by self-attestation.

---

## 3. Design

### 3.1 Three-state model (evidence-derived)

Each node result carries a `verification` object. Status is **derived deterministically**:

```
verified    ← has REAL external evidence: verification.method ≠ 'none',
              evidence_path exists on disk, verifier ≠ generator
unverified  ← generated but no real evidence (method 'none' or evidence missing)
failed      ← blocking findings / hard_fail
```

`unverified` is **never** counted toward the completeness numerator.

### 3.2 Two-column completeness report (cannot lie)

`compute_completeness.py` reads `workflow.json` transition_log (each entry now carries `verification`) and emits:

```
verified   (真验过):      28%   ← the headline number
unverified (只生成没验):   67%   ← surfaced loudly, NOT in the numerator
failed:                    5%
```

Had TeteChat reported this way, the user would have seen **"verified ~28%"** — matching the real < 30% — instead of 94.65.

### 3.3 Two enforcement layers

**Deterministic (the keystone — kills lazy inflation, unit-tested):**
- `check_evidence.py`: any node claiming `verified` MUST have a non-empty `evidence_path` that **exists on disk** and a `verifier` distinct from the generator. Missing → **downgrade to `unverified`**. This makes the *default lazy path* (just write a PASS report) score `unverified`, not 94.65 — regardless of LLM behavior.
- `compute_completeness.py`: the two-column report.

**LLM adversarial (catches fabricated evidence + hollow code):**
- **Independent verifier protocol** (`verification-protocol.md`): the verify agent is NOT the generate agent, must *run the real thing*, capture evidence, and is prompted to **disprove** "done."
- **Hollowness detector** (`hollowness-detector.md` capability): an adversarial agent that hunts fake-success / mock-backend / UI-with-no-real-data / handler-returns-200-with-stub, and **downgrades** the affected node's status + records findings.

### 3.4 Honest boundary (no overclaim)

The deterministic gate guarantees evidence **exists and is the right type** — it cannot stop an agent that **fabricates** evidence (a blank screenshot, an invented command transcript). So:
- The deterministic layer makes the **lazy default honest** (≈90% of the inflation).
- The LLM adversarial layer checks evidence **authenticity** + code hollowness.
- The transparent two-column report lets a **human spot-check**.

We **cannot** make lying impossible; we make the default path honest, fabrication catchable, and the report transparent. That is the difference between "94.65 lies" and "28% doesn't."

### 3.5 Launch-readiness gate

A node serving a **critical user flow** (from product-map task-inventory) must reach `verified` (real evidence) before the product is called launch-ready. `unverified` critical flows block the launch claim, not the run.

## 4. Data contracts

```jsonc
// NODE_RESULT_SCHEMA gains:
"verification": {
  "method": "real-run | real-test | real-api | db-query | screenshot | none",
  "evidence_path": "string  // captured proof file; must exist for 'verified'",
  "verifier": "string       // agent identity; must differ from the generator",
  "claim": "string          // one line: what was proven to work"
}
```

```jsonc
// completeness-report.json (compute_completeness.py output)
{
  "total": 152, "verified": 43, "unverified": 102, "failed": 7,
  "verified_pct": 28.3, "unverified_pct": 67.1, "failed_pct": 4.6,
  "by_node": [{ "node_id": "...", "state": "verified|unverified|failed", "method": "...", "evidence_path": "..." }],
  "critical_unverified": ["...node_ids serving critical flows but lacking evidence..."]
}
```

```jsonc
// hollowness-report.json (hollowness-detector capability output)
{ "hollow_nodes": [{ "node_id": "...", "type": "fake_success|mocked_backend|ui_no_data|stub_handler", "evidence": "...", "downgrade_to": "unverified" }] }
```

## 6. Locked decisions

1. Epistemic honesty: "generated" never counts as "verified."
2. Three-state (verified/unverified/failed), **evidence-derived** and deterministic.
3. Evidence requirement is the keystone — enforced by `check_evidence.py`, not LLM goodwill.
4. Two-column completeness report; `unverified` never in the numerator.
5. Hybrid: deterministic gate (lazy-honest) + LLM adversarial (authenticity + hollowness).
6. Honest boundary stated: not un-gameable; default-honest + fabrication-catchable + transparent.
7. Launch gate: critical flows require real evidence.
8. Full overhaul scope: core + protocol + hollowness detector + verify-capability rewrite to the protocol.
